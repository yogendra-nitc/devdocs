Q1

What is Spring Boot and how is it different from the traditional Spring Framework?

Core ▼

The traditional Spring Framework is extremely powerful but requires a lot of manual setup — you had to write XML configuration files or Java config classes for every bean, set up a web server, configure the DispatcherServlet, manage dependency versions, and wire everything together yourself. Spring Boot is an opinionated layer on top of Spring that eliminates most of that boilerplate through three core ideas: auto-configuration (Spring Boot looks at your classpath and automatically configures beans you would otherwise define manually), starter dependencies (curated groups of compatible dependencies bundled under a single artifact like spring-boot-starter-web), and an embedded server (Tomcat, Jetty, or Undertow is bundled inside the JAR so you run the app with a plain java -jar command). The result is production-ready apps with minimal configuration and zero XML.

Java 
// Traditional Spring — manual DispatcherServlet setup in web.xml + applicationContext.xml
// Dozens of XML lines just to start a web app.

// Spring Boot — this is the ENTIRE entry point of a web application:
@SpringBootApplication          // auto-config + component scan + config
public class MyApp {
    public static void main(String[] args) {
        SpringApplication.run(MyApp.class, args);
        // Embedded Tomcat starts, beans wired, app ready — no server install needed
    }
}

// pom.xml — one starter pulls in Spring MVC + embedded Tomcat + JSON support
// <dependency>
//   <groupId>org.springframework.boot</groupId>
//   <artifactId>spring-boot-starter-web</artifactId>
// </dependency>
Q2

What does @SpringBootApplication do? What three annotations does it combine?

Core ▼

@SpringBootApplication is a convenience meta-annotation that combines three annotations into one. @SpringBootConfiguration marks the class as a source of Spring bean definitions (it is itself a specialisation of @Configuration). @EnableAutoConfiguration tells Spring Boot to automatically configure beans based on the JARs present on the classpath — for example, if spring-boot-starter-web is on the classpath, Spring Boot auto-configures a DispatcherServlet and an embedded Tomcat. @ComponentScan tells Spring to scan the current package and all sub-packages for classes annotated with @Component, @Service, @Repository, @Controller, etc. and register them as beans. Together these three annotations bootstrap the entire application context.

Java 
// What @SpringBootApplication expands to internally:
@SpringBootConfiguration   // = @Configuration — this class defines beans
@EnableAutoConfiguration   // classpath-based auto-config magic
@ComponentScan             // scan com.example and all sub-packages
public class Application { }

// You can customise the scan base packages:
@SpringBootApplication(scanBasePackages = {"com.myapp.web", "com.myapp.service"})
public class Application { }

// Or exclude a specific auto-configuration:
@SpringBootApplication(exclude = { DataSourceAutoConfiguration.class })
public class Application { }
Q3

What is Spring Boot Auto-Configuration and how does it work under the hood?

Core ▼

Auto-configuration is the mechanism that automatically creates and registers Spring beans based on what is present on the classpath and what properties are set. When @EnableAutoConfiguration is active, Spring Boot loads a list of candidate auto-configuration classes from META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports (Spring Boot 3.x) or spring.factories (Spring Boot 2.x). Each candidate class is annotated with conditions like @ConditionalOnClass, @ConditionalOnMissingBean, and @ConditionalOnProperty. Only those whose conditions evaluate to true are applied. For example, DataSourceAutoConfiguration fires only if a JDBC driver class is on the classpath AND no DataSource bean is already defined. Your own beans always take precedence over auto-configured ones — you never lose control.

Java 
// How an auto-configuration class looks (simplified):
@AutoConfiguration
@ConditionalOnClass({ DataSource.class, EmbeddedDatabaseType.class })
// ↑ only active if these classes exist on the classpath
@ConditionalOnMissingBean(DataSource.class)
// ↑ backs off if the developer already defined their own DataSource
public class DataSourceAutoConfiguration {

    @Bean
    public DataSource dataSource() {
        return /* create embedded H2 DataSource */;
    }
}

// See all auto-config decisions at startup (very useful for debugging):
// Set in application.properties:
// logging.level.org.springframework.boot.autoconfigure=DEBUG
// Or run with --debug flag: java -jar myapp.jar --debug
Q4

What is Dependency Injection in Spring? What are the three types and which is recommended?

Core ▼

Dependency Injection (DI) is a design pattern where an object's dependencies are provided by the framework rather than created by the object itself. Spring manages this through its IoC (Inversion of Control) container. There are three ways to inject dependencies. Constructor injection passes dependencies through the constructor — the recommended approach because it makes dependencies explicit, supports immutability (fields can be final), and makes testing easy (you simply pass mock objects). Setter injection uses setter methods — suitable for optional dependencies. Field injection uses @Autowired directly on a field — convenient but considered bad practice because it hides dependencies, makes the class harder to test, and requires reflection. Spring's own documentation recommends constructor injection for mandatory dependencies.

Java 
// ✅ Constructor Injection — RECOMMENDED
@Service
public class OrderService {
    private final PaymentService paymentService;   // final — immutable
    private final InventoryService inventoryService;

    // @Autowired is optional when there's only one constructor (Spring 4.3+)
    public OrderService(PaymentService ps, InventoryService is) {
        this.paymentService   = ps;
        this.inventoryService = is;
    }
}

// ⚠️  Setter Injection — for optional dependencies
@Autowired(required = false)
public void setEmailService(EmailService emailService) {
    this.emailService = emailService;
}

// ❌ Field Injection — avoid, hides dependencies, hard to test
@Autowired
private PaymentService paymentService;
Q5

What is a Spring Bean? What are the different bean scopes available?

Core ▼

A Spring Bean is any object that is managed by the Spring IoC container — created, wired, and destroyed by Spring rather than by your code using new. You declare beans using @Component (and its specialisations @Service, @Repository, @Controller) or with @Bean inside a @Configuration class. Every bean has a scope that controls how many instances Spring creates and how they are shared. Singleton (default) — one instance per application context, shared by all. Prototype — a new instance every time the bean is requested. Request — one instance per HTTP request (web apps only). Session — one instance per HTTP session. Application — one instance per ServletContext. Singleton is the right choice for stateless services; prototype is for stateful objects that must not be shared.

Java 
// Singleton scope (default) — one shared instance
@Service                          // implicitly @Scope("singleton")
public class EmailService { }

// Prototype scope — new object every time it's injected/requested
@Component
@Scope("prototype")
public class ReportBuilder { }

// Request scope — one per HTTP request
@Component
@Scope(value = WebApplicationContext.SCOPE_REQUEST,
       proxyMode = ScopedProxyMode.TARGET_CLASS)
public class RequestContext { }

// @Bean in a @Configuration class
@Configuration
public class AppConfig {
    @Bean
    @Scope("prototype")
    public ReportBuilder reportBuilder() {
        return new ReportBuilder();
    }
}
Q6

What is the Spring Application Context? How is it different from the BeanFactory?

Core ▼

BeanFactory is the basic IoC container — it can instantiate and wire beans on demand (lazy loading). ApplicationContext extends BeanFactory and adds enterprise-level features: it pre-instantiates all singleton beans at startup (eager loading), supports internationalization via MessageSource, publishes application events to listeners, supports AOP integration, and provides environment/property abstraction. In Spring Boot, the application context is created automatically when SpringApplication.run() is called. For web apps it creates a AnnotationConfigServletWebServerApplicationContext which also starts the embedded web server. You should almost always use ApplicationContext — BeanFactory is only relevant in extremely resource-constrained environments.

Java 
// Spring Boot creates and manages the ApplicationContext for you
@SpringBootApplication
public class MyApp {
    public static void main(String[] args) {
        ConfigurableApplicationContext ctx =
            SpringApplication.run(MyApp.class, args);

        // Access beans programmatically if needed
        OrderService svc = ctx.getBean(OrderService.class);

        // Check if a bean exists
        boolean exists = ctx.containsBean("orderService");

        // Publish a custom event to all listeners
        ctx.publishEvent(new OrderCreatedEvent("ORD-001"));
    }
}

// Implementing ApplicationContextAware to access context inside a bean
@Component
public class BeanInspector implements ApplicationContextAware {
    private ApplicationContext context;

    @Override
    public void setApplicationContext(ApplicationContext ctx) {
        this.context = ctx;
    }
}
Q7

What is the difference between @Component, @Service, @Repository, and @Controller?

Core ▼

All four are specialisations of @Component — at the technical level, they all make a class eligible for component scanning and register it as a Spring bean. The difference is semantic and functional. @Component is a generic stereotype for any Spring-managed component. @Service marks the service/business-logic layer — it has no extra behaviour today but signals intent, making code easier to understand. @Repository marks the data-access layer and additionally enables automatic exception translation — Spring wraps low-level data-access exceptions (like JDBC's SQLException) into Spring's consistent DataAccessException hierarchy. @Controller marks a class as an MVC web controller that handles HTTP requests and returns view names. @RestController combines @Controller and @ResponseBody, making every method return data directly as JSON/XML rather than a view name.

Java 
@Component          // generic — use when no other stereotype fits
public class PasswordEncoder { }

@Service            // business logic layer
public class OrderService {
    public Order placeOrder(OrderRequest req) { return new Order(); }
}

@Repository         // data-access layer + exception translation
public class OrderRepository {
    public Order findById(long id) { return /* DB call */ null; }
    // SQLException here → auto-translated to DataAccessException
}

@Controller         // returns view names (Thymeleaf/JSP)
public class HomeController {
    @GetMapping("/")
    public String home() { return "index"; }  // resolves to index.html
}

@RestController     // @Controller + @ResponseBody — returns JSON directly
public class ApiController {
    @GetMapping("/api/orders")
    public List<Order> orders() { return List.of(); }  // serialised to JSON
}
Q8

What is Spring AOP? What problems does it solve and what are key AOP concepts?

Core ▼

AOP (Aspect-Oriented Programming) solves the problem of cross-cutting concerns — behaviour that spans many layers of an application but doesn't belong in any single class: logging, transaction management, security checks, performance monitoring, and caching. Without AOP, this code gets copy-pasted into every service method. Key concepts: a Join Point is a specific point in execution (a method call). A Pointcut is an expression that matches a set of join points. An Advice is the code to run at matched join points (@Before, @After, @Around, @AfterReturning, @AfterThrowing). An Aspect is a class that groups pointcuts and advice. Spring AOP works via dynamic proxies — it wraps your bean in a proxy that intercepts calls. This means AOP only intercepts calls that go through Spring's proxy, not internal this.method() calls.

Java 
@Aspect
@Component
public class LoggingAspect {

    // Pointcut: matches all methods in the service package
    @Pointcut("execution(* com.example.service.*.*(..))")
    public void serviceMethods() { }

    // @Before advice — runs BEFORE the matched method
    @Before("serviceMethods()")
    public void logBefore(JoinPoint jp) {
        System.out.println("Calling: " + jp.getSignature().getName());
    }

    // @Around advice — wraps the method, controls execution
    @Around("serviceMethods()")
    public Object measureTime(ProceedingJoinPoint pjp) throws Throwable {
        long start = System.currentTimeMillis();
        Object result = pjp.proceed();   // invoke actual method
        long elapsed = System.currentTimeMillis() - start;
        System.out.println("Took: " + elapsed + "ms");
        return result;
    }

    // @AfterThrowing — runs if the method throws an exception
    @AfterThrowing(pointcut = "serviceMethods()", throwing = "ex")
    public void logError(Exception ex) {
        System.out.println("Exception: " + ex.getMessage());
    }
}
Q9

What is application.properties vs application.yml? What is the difference and which should you use?

Configuration ▼

Both files serve the same purpose — externalising configuration — and Spring Boot supports both equally. application.properties uses a flat key=value format, which is simple and widely familiar. application.yml uses YAML's hierarchical indented format, which is more readable for deeply nested properties because you avoid repeating the common prefix. For example, three spring.datasource.* properties in .properties format become a single indented datasource: block in YAML. YAML also supports lists more naturally. The downside of YAML is that indentation errors cause subtle bugs and it cannot be easily concatenated the way multiple .properties files can. Choose based on team preference — many projects use .yml for the clean hierarchy it provides.

Java 
// --- application.properties (flat key=value) ---
// server.port=8081
// spring.datasource.url=jdbc:mysql://localhost:3306/mydb
// spring.datasource.username=root
// spring.datasource.password=secret
// spring.jpa.hibernate.ddl-auto=update

// --- application.yml (hierarchical, less repetition) ---
// server:
//   port: 8081
// spring:
//   datasource:
//     url: jdbc:mysql://localhost:3306/mydb
//     username: root
//     password: secret
//   jpa:
//     hibernate:
//       ddl-auto: update

// Reading properties in Java using @Value
@Component
public class AppConfig {
    @Value("${server.port}")
    private int serverPort;

    @Value("${app.name:MyDefaultApp}")  // :MyDefaultApp is the fallback
    private String appName;
}
Q10

What is @ConfigurationProperties? How is it better than @Value for binding configuration?

Configuration ▼

@Value injects one property at a time and requires you to repeat the full key string in every class that uses it. If a prefix changes, you update it everywhere. @ConfigurationProperties binds an entire group of related properties to a POJO at once using a shared prefix. All fields in the class are automatically mapped by name. This gives you type safety (the IDE validates property names), supports complex types like List, Map, and nested objects, enables IDE auto-completion in .properties/.yml files (with the metadata processor), and makes the configuration self-documenting. It also works well with @Validated for bean-validation constraints on properties.

Java 
// application.yml
// app:
//   name: MyShop
//   max-retries: 3
//   allowed-origins:
//     - https://frontend.com
//     - https://admin.com

// @ConfigurationProperties POJO — all fields bound automatically
@ConfigurationProperties(prefix = "app")
@Validated
public class AppProperties {

    @NotBlank
    private String name;

    @Min(1) @Max(10)
    private int maxRetries;

    private List<String> allowedOrigins;

    // getters and setters (or use Lombok @Data)
}

// Enable it in your main class or config:
@SpringBootApplication
@EnableConfigurationProperties(AppProperties.class)
public class Application { }

// Inject and use it anywhere:
@Service
public class StartupService {
    private final AppProperties props;

    public StartupService(AppProperties props) { this.props = props; }

    public void start() {
        System.out.println("Starting: " + props.getName());
    }
}
Q11

What are Spring Profiles? How do you activate them and use them for environment-specific configuration?

Configuration ▼

Spring Profiles allow you to segregate parts of your application configuration and make them available only in certain environments — dev, test, staging, prod. You define profile-specific property files as application-{profile}.yml (e.g., application-dev.yml, application-prod.yml). At startup, the active profile's properties override the defaults in application.yml. You can also annotate beans with @Profile("dev") so they are only registered when that profile is active — useful for swapping out mock services for real ones. Profiles are activated via spring.profiles.active in the property file, as a JVM argument -Dspring.profiles.active=prod, or as an environment variable SPRING_PROFILES_ACTIVE=prod.

Java 
// application.yml — base (all environments)
// app.name: MyApp
// server.port: 8080

// application-dev.yml — overrides for dev
// server.port: 9090
// logging.level.com.example: DEBUG

// application-prod.yml — overrides for prod
// server.port: 443
// logging.level.com.example: WARN

// Activate dev profile (3 ways):
// 1. application.yml: spring.profiles.active: dev
// 2. JVM arg: java -Dspring.profiles.active=prod -jar app.jar
// 3. Env var: export SPRING_PROFILES_ACTIVE=prod

// Profile-specific beans — only registered in 'dev' profile
@Component
@Profile("dev")
public class MockEmailService implements EmailService {
    public void send(String to, String body) {
        System.out.println("[DEV] Fake email to: " + to);
    }
}

@Component
@Profile("prod")
public class SmtpEmailService implements EmailService {
    public void send(String to, String body) { /* real SMTP */ }
}
Q12

What is the Spring Boot Actuator? What does it provide out of the box?

Configuration ▼

Spring Boot Actuator adds production-ready monitoring and management endpoints to your application without writing any code. Add the spring-boot-starter-actuator dependency and you get HTTP endpoints (and JMX) that expose operational information. Key endpoints: /actuator/health — reports if the app is UP or DOWN (and details about DB, disk, cache). /actuator/info — app metadata (version, build). /actuator/metrics — JVM memory, CPU, HTTP request counts, etc. /actuator/env — all configuration properties. /actuator/beans — all registered Spring beans. /actuator/loggers — view and dynamically change log levels at runtime. Most endpoints are disabled by default for security — you explicitly expose them via management.endpoints.web.exposure.include.

Java 
// pom.xml dependency:
// <dependency>
//   <groupId>org.springframework.boot</groupId>
//   <artifactId>spring-boot-starter-actuator</artifactId>
// </dependency>

// application.yml — expose specific endpoints
// management:
//   endpoints:
//     web:
//       exposure:
//         include: health,info,metrics,loggers
//   endpoint:
//     health:
//       show-details: always

// Custom Health Indicator — plugs into /actuator/health
@Component
public class ExternalApiHealthIndicator implements HealthIndicator {
    @Override
    public Health health() {
        boolean apiReachable = pingExternalApi();
        return apiReachable
            ? Health.up().withDetail("api", "reachable").build()
            : Health.down().withDetail("api", "unreachable").build();
    }
    private boolean pingExternalApi() { return true; /* real check */ }
}

// GET /actuator/health → {"status":"UP","components":{"externalApi":{"status":"UP"}}}
Q13

What is the Spring Boot DevTools and what does it offer during development?

Configuration ▼

Spring Boot DevTools is a development-time module that makes your development loop faster. Its headline feature is automatic restart — whenever a class on the classpath changes, DevTools restarts the application context in milliseconds by using two classloaders: a base classloader for third-party libraries (rarely changes) and a restart classloader for your classes (changes often). Only the restart classloader is reloaded, making restarts much faster than a cold start. DevTools also enables LiveReload — a browser extension that refreshes the page automatically when static resources (HTML, CSS, JS) change. Additionally, it changes several defaults suitable for dev: disables template caching (so Thymeleaf changes are visible immediately), sets the logging level to DEBUG, and enables H2 console. DevTools is automatically disabled when running a packaged JAR in production.

Java 
// pom.xml — add DevTools (automatically excluded from production JAR)
// <dependency>
//   <groupId>org.springframework.boot</groupId>
//   <artifactId>spring-boot-devtools</artifactId>
//   <optional>true</optional>   ← not inherited by downstream projects
// </dependency>

// application.properties — DevTools sets these defaults automatically in dev:
// spring.thymeleaf.cache=false         (see template changes immediately)
// spring.web.resources.cache.period=0  (no static resource caching)
// logging.level.web=DEBUG

// Customise which paths trigger a restart:
// spring.devtools.restart.additional-paths=src/main/resources/templates
// spring.devtools.restart.exclude=static/**,public/**

// Trigger file — restart only when this file changes (useful in IntelliJ)
// spring.devtools.restart.trigger-file=.reloadtrigger

// DevTools is DISABLED when:
// - Running a fully packaged JAR: java -jar myapp.jar
// - When spring.devtools.restart.enabled=false is set
// - System property: spring.devtools.restart.enabled=false
Q14

What are the @Conditional annotations? How do they control bean registration?

Configuration ▼

@Conditional annotations tell Spring to register a bean only when a specific condition is true at application startup. They are the backbone of auto-configuration. Common ones: @ConditionalOnClass — register only if a specific class is on the classpath (e.g., only configure Redis if Jedis is present). @ConditionalOnMissingBean — register only if no bean of that type is already defined (lets user override the default). @ConditionalOnProperty — register only if a property has a specific value. @ConditionalOnMissingClass — opposite of OnClass. @ConditionalOnBean — register only if another specific bean already exists. These annotations let you write smart, adaptive configuration that responds to the runtime environment.

Java 
// Register CacheService only if Caffeine library is on the classpath
@Bean
@ConditionalOnClass(name = "com.github.benmanes.caffeine.cache.Caffeine")
public CacheService caffeineCacheService() {
    return new CaffeineCacheService();
}

// Provide a default bean only if the developer hasn't defined their own
@Bean
@ConditionalOnMissingBean(EmailService.class)
public EmailService defaultEmailService() {
    return new ConsoleEmailService();
}

// Register a bean only when a property is set to a specific value
@Bean
@ConditionalOnProperty(name = "feature.dark-mode", havingValue = "true",
                        matchIfMissing = false)
public DarkModeService darkModeService() {
    return new DarkModeService();
}

// Custom condition — implement Condition interface
public class LinuxCondition implements Condition {
    @Override
    public boolean matches(ConditionContext ctx, AnnotatedTypeMetadata meta) {
        return System.getProperty("os.name").contains("Linux");
    }
}

@Bean
@Conditional(LinuxCondition.class)
public LinuxFileWatcher fileWatcher() { return new LinuxFileWatcher(); }
Q15

What is the difference between @Bean and @Component? When do you use each?

Configuration ▼

@Component is a class-level annotation — you put it on a class you own and control the source code of. Spring detects it via component scanning and registers an instance as a bean. @Bean is a method-level annotation placed inside a @Configuration class — the method's return value becomes the bean. You use @Bean when: (1) you need to register a third-party class as a bean (you cannot add @Component to code you don't own — e.g., an ObjectMapper from Jackson), (2) you need complex construction logic (builder patterns, conditional setup), or (3) you want to control the bean name, scope, or init/destroy methods explicitly. @Component is simpler and preferred for your own classes.

Java 
// @Component — on a class YOU own. Spring detects via component scan.
@Service
public class OrderService {
    public Order process(OrderRequest req) { return new Order(); }
}

// @Bean — for third-party classes or complex wiring
@Configuration
public class InfraConfig {

    // Jackson's ObjectMapper — can't add @Component to it
    @Bean
    public ObjectMapper objectMapper() {
        return new ObjectMapper()
            .enable(SerializationFeature.INDENT_OUTPUT)
            .disable(DeserializationFeature.FAIL_ON_UNKNOWN_PROPERTIES);
    }

    // RestTemplate — third-party, needs custom config
    @Bean
    public RestTemplate restTemplate() {
        RestTemplate rt = new RestTemplate();
        rt.setErrorHandler(new CustomErrorHandler());
        return rt;
    }

    // Named bean, custom init/destroy
    @Bean(name = "primaryCache", initMethod = "init", destroyMethod = "shutdown")
    public CacheManager cacheManager() {
        return new RedisCacheManager();
    }
}
Q16

What is a RESTful API? How does Spring Boot support building one?

REST ▼

REST (Representational State Transfer) is an architectural style for designing networked APIs. The core principles are: statelessness (each request is self-contained), resource identification via URLs, HTTP verbs that represent actions (GET = read, POST = create, PUT = full update, PATCH = partial update, DELETE = remove), and uniform response formats (typically JSON). Spring Boot supports REST APIs through Spring MVC: @RestController marks a class whose methods return response bodies directly (not view names). @RequestMapping and its shortcuts (@GetMapping, @PostMapping, @PutMapping, @DeleteMapping) map HTTP requests to methods. Jackson auto-serializes return objects to JSON without any configuration.

Java 
@RestController                          // @Controller + @ResponseBody
@RequestMapping("/api/products")          // base URL for all methods
public class ProductController {

    private final ProductService productService;

    public ProductController(ProductService ps) { this.productService = ps; }

    @GetMapping                              // GET /api/products
    public List<Product> getAll() {
        return productService.findAll();      // serialised to JSON array
    }

    @GetMapping("/{id}")                   // GET /api/products/42
    public Product getById(@PathVariable Long id) {
        return productService.findById(id);
    }

    @PostMapping                             // POST /api/products
    @ResponseStatus(HttpStatus.CREATED)      // returns 201
    public Product create(@RequestBody @Valid Product product) {
        return productService.save(product);
    }

    @DeleteMapping("/{id}")               // DELETE /api/products/42
    @ResponseStatus(HttpStatus.NO_CONTENT)  // returns 204
    public void delete(@PathVariable Long id) {
        productService.delete(id);
    }
}
Q17

What is the difference between @PathVariable, @RequestParam, and @RequestBody?

REST ▼

These three annotations extract different parts of an HTTP request. @PathVariable extracts a value embedded in the URL path — it is part of the URL structure itself (e.g., /users/42 where 42 identifies the resource). @RequestParam extracts a query parameter from the URL's query string (e.g., /users?page=2&size=10). It can be made optional with a default value. @RequestBody deserialises the entire HTTP request body (typically JSON) into a Java object using Jackson. It is used with POST/PUT requests where you send structured data. A single method can use all three together.

Java 
@RestController
@RequestMapping("/api/users")
public class UserController {

    // @PathVariable — value is part of the URL path
    // GET /api/users/42
    @GetMapping("/{id}")
    public User getById(@PathVariable Long id) {
        return userService.findById(id);
    }

    // @RequestParam — query parameters after '?'
    // GET /api/users?page=0&size=10&sort=name
    @GetMapping
    public Page<User> getAll(
            @RequestParam(defaultValue = "0")   int page,
            @RequestParam(defaultValue = "10")  int size,
            @RequestParam(required = false)     String sort) {
        return userService.findAll(page, size, sort);
    }

    // @RequestBody — deserialises JSON body into a Java object
    // POST /api/users   with body: {"name":"Alice","email":"alice@example.com"}
    @PostMapping
    public User create(@RequestBody @Valid CreateUserRequest request) {
        return userService.create(request);
    }

    // All three combined: PUT /api/users/42?notify=true + JSON body
    @PutMapping("/{id}")
    public User update(@PathVariable Long id,
                        @RequestParam(defaultValue = "false") boolean notify,
                        @RequestBody UpdateUserRequest req) {
        return userService.update(id, req, notify);
    }
}
Q18

What is ResponseEntity? Why is it useful and when should you use it?

REST ▼

ResponseEntity<T> gives you full control over the HTTP response — the status code, headers, and body — in a single return value. When you return a plain object from a controller method, Spring always uses 200 OK and no custom headers. But REST APIs need to return 201 Created with a Location header after creation, 404 Not Found when a resource is missing, 204 No Content on deletion, and custom headers for pagination, rate limiting, etc. ResponseEntity makes all of this possible without needing HttpServletResponse. It is particularly important for building APIs that follow HTTP semantics properly.

Java 
@RestController
@RequestMapping("/api/orders")
public class OrderController {

    // Return 200 OK with body — ResponseEntity wraps it
    @GetMapping("/{id}")
    public ResponseEntity<Order> getOrder(@PathVariable Long id) {
        return orderService.findById(id)
            .map(ResponseEntity::ok)                          // 200 if found
            .orElse(ResponseEntity.notFound().build());       // 404 if not
    }

    // 201 Created with Location header pointing to new resource
    @PostMapping
    public ResponseEntity<Order> create(@RequestBody OrderRequest req,
                                           UriComponentsBuilder ucb) {
        Order saved = orderService.create(req);
        URI location = ucb.path("/api/orders/{id}")
                          .buildAndExpand(saved.getId()).toUri();
        return ResponseEntity.created(location).body(saved);   // 201 + Location
    }

    // 204 No Content on delete
    @DeleteMapping("/{id}")
    public ResponseEntity<Void> delete(@PathVariable Long id) {
        orderService.delete(id);
        return ResponseEntity.noContent().build();             // 204
    }

    // Custom headers — e.g., rate limit info
    @GetMapping
    public ResponseEntity<List<Order>> getAll() {
        return ResponseEntity.ok()
            .header("X-Total-Count", "250")
            .header("X-Rate-Limit-Remaining", "99")
            .body(orderService.findAll());
    }
}
Q19

What is global exception handling in Spring Boot? How do you implement it with @ControllerAdvice?

REST ▼

Without global exception handling, every controller method needs its own try-catch block, exceptions bubble up as ugly 500 errors, and there is no consistent error response format across the API. @ControllerAdvice (or @RestControllerAdvice for REST) lets you define a single class that handles exceptions thrown from any controller in the application. Inside it, @ExceptionHandler methods are matched to specific exception types. This keeps your controllers clean, gives you a consistent error response structure (with timestamp, status, message, path), and lets you return the correct HTTP status code for each type of error. It is one of the most important patterns in production Spring Boot APIs.

Java 
// Standard error response record
public record ApiError(
    int    status,
    String message,
    String path,
    String timestamp
) { }

// Global exception handler — applies to ALL controllers
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(ResourceNotFoundException.class)
    @ResponseStatus(HttpStatus.NOT_FOUND)
    public ApiError handleNotFound(ResourceNotFoundException ex,
                                       HttpServletRequest req) {
        return new ApiError(404, ex.getMessage(), req.getRequestURI(),
                             LocalDateTime.now().toString());
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    public Map<String, String> handleValidation(MethodArgumentNotValidException ex) {
        Map<String, String> errors = new HashMap<>();
        ex.getBindingResult().getFieldErrors()
          .forEach(e -> errors.put(e.getField(), e.getDefaultMessage()));
        return errors;  // {"email": "must be valid", "name": "must not be blank"}
    }

    @ExceptionHandler(Exception.class)
    @ResponseStatus(HttpStatus.INTERNAL_SERVER_ERROR)
    public ApiError handleGeneric(Exception ex, HttpServletRequest req) {
        return new ApiError(500, "Internal error", req.getRequestURI(),
                             LocalDateTime.now().toString());
    }
}
Q20

What is bean validation in Spring Boot? How do you use @Valid and validation annotations?

REST ▼

Bean Validation (JSR-380, implemented by Hibernate Validator) lets you declare constraints on your model fields using annotations and have Spring automatically validate incoming request objects. Add spring-boot-starter-validation to enable it. Annotate your request DTO with constraint annotations like @NotNull, @NotBlank, @Email, @Min, @Max, @Size, @Pattern. Then add @Valid on the @RequestBody parameter in your controller. If validation fails, Spring throws MethodArgumentNotValidException, which you can handle in @ControllerAdvice to return detailed error messages. You can also create custom validators for domain-specific rules.

Java 
// Request DTO with validation constraints
public class CreateUserRequest {

    @NotBlank(message = "Name is required")
    @Size(min = 2, max = 50, message = "Name must be 2-50 characters")
    private String name;

    @Email(message = "Must be a valid email address")
    @NotBlank
    private String email;

    @Min(value = 18, message = "Age must be at least 18")
    @Max(value = 120, message = "Age must be realistic")
    private int age;

    @Pattern(regexp = "^\\+?[0-9]{10,15}$", message = "Invalid phone number")
    private String phone;
}

// Controller — @Valid triggers validation before the method body runs
@PostMapping("/api/users")
public ResponseEntity<User> create(@RequestBody @Valid CreateUserRequest req) {
    // only reached if ALL constraints pass
    return ResponseEntity.ok(userService.create(req));
}

// Validation on method parameters (service layer) — use @Validated on the class
@Service
@Validated
public class ProductService {
    public Product findById(@Positive(message = "ID must be positive") Long id) {
        return repo.findById(id).orElseThrow();
    }
}
Q21

What is RestTemplate and WebClient? Which should you use in a modern Spring Boot application?

REST ▼

Both are HTTP clients for calling external REST APIs from your Spring Boot app. RestTemplate is the older, synchronous (blocking) client — the calling thread waits until the HTTP response arrives. It is simple to use and still fully supported but is in maintenance mode as of Spring 5. WebClient (from Spring WebFlux) is the modern, non-blocking, reactive client — it can execute requests asynchronously without tying up a thread. Even in a traditional (non-reactive) Spring MVC app, you can use WebClient in blocking mode via .block(). For new projects, Spring recommends WebClient. In Spring Boot 3.2+, RestClient was introduced as a modern synchronous client with a fluent API similar to WebClient but blocking — a good middle ground.

Java 
// ── RestTemplate (older, blocking, still widely used) ──────────────────
@Bean
public RestTemplate restTemplate() { return new RestTemplate(); }

@Service
public class WeatherService {
    private final RestTemplate restTemplate;

    public WeatherData getWeather(String city) {
        String url = "https://api.weather.com/v1/city?name=" + city;
        return restTemplate.getForObject(url, WeatherData.class); // blocks
    }
}

// ── WebClient (modern, non-blocking, recommended) ──────────────────────
@Bean
public WebClient webClient() {
    return WebClient.builder()
        .baseUrl("https://api.weather.com")
        .defaultHeader(HttpHeaders.CONTENT_TYPE, MediaType.APPLICATION_JSON_VALUE)
        .build();
}

public Mono<WeatherData> getWeatherAsync(String city) {
    return webClient.get()
        .uri("/v1/city?name={city}", city)
        .retrieve()
        .bodyToMono(WeatherData.class);  // non-blocking
}

// Use .block() for synchronous execution in a non-reactive app:
WeatherData data = getWeatherAsync("London").block();
Q22

What is CORS and how do you configure it in Spring Boot?

REST ▼

CORS (Cross-Origin Resource Sharing) is a browser security mechanism that blocks JavaScript running on one origin (e.g., https://frontend.com) from making HTTP requests to a different origin (e.g., https://api.myapp.com). The browser sends a preflight OPTIONS request to ask the server if the cross-origin call is allowed. The server must respond with the correct Access-Control-Allow-* headers. Spring Boot provides three ways to configure this: @CrossOrigin on individual controller methods or classes (for targeted control), a global WebMvcConfigurer bean (for app-wide policy), or through Spring Security's CORS configuration (when security is involved).

Java 
// Method 1: @CrossOrigin on a controller
@RestController
@CrossOrigin(origins = "https://frontend.com", maxAge = 3600)
public class ApiController {

    // More granular: only this endpoint allows any origin
    @CrossOrigin(origins = "*")
    @GetMapping("/public")
    public String publicEndpoint() { return "open"; }
}

// Method 2: Global CORS config — applies to all controllers
@Configuration
public class WebConfig implements WebMvcConfigurer {

    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**")
                .allowedOrigins("https://frontend.com", "https://admin.com")
                .allowedMethods("GET", "POST", "PUT", "DELETE")
                .allowedHeaders("*")
                .allowCredentials(true)
                .maxAge(3600);
    }
}
Q23

What is Spring Transaction Management? How does @Transactional work?

Data ▼

A transaction is a unit of work that either fully succeeds or fully rolls back — the classic bank transfer: debit one account AND credit another must both succeed or neither should. @Transactional is Spring's declarative way to wrap a method in a transaction. Spring uses AOP to intercept the method call, begin a transaction before it runs, and either commit (if the method completes normally) or roll back (if an unchecked exception is thrown). By default, Spring only rolls back on unchecked exceptions (RuntimeException and Error) — NOT on checked exceptions. Key attributes: rollbackFor, propagation (how the transaction behaves when a transactional method calls another), and isolation (protection level against concurrent transactions).

Java 
@Service
public class BankService {

    // Default: commits on success, rolls back on RuntimeException
    @Transactional
    public void transfer(Long fromId, Long toId, BigDecimal amount) {
        accountRepo.debit(fromId, amount);    // step 1
        accountRepo.credit(toId, amount);   // step 2
        // If step 2 throws RuntimeException → step 1 is rolled back
    }

    // Roll back on checked exceptions too
    @Transactional(rollbackFor = Exception.class)
    public void processOrder(Order order) throws OrderException {
        /* ... */
    }

    // Read-only transaction — performance optimisation for queries
    @Transactional(readOnly = true)
    public List<Account> findAll() { return accountRepo.findAll(); }

    // Propagation — REQUIRES_NEW starts a fresh, independent transaction
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void auditLog(String action) {
        // runs in its own transaction, committed independently
    }
}
Q24

What is Spring Caching? How do you use @Cacheable, @CachePut, and @CacheEvict?

Data ▼

Spring's caching abstraction lets you cache method return values so expensive operations (database queries, remote API calls) are not repeated unnecessarily. Enable it with @EnableCaching on your config class. @Cacheable on a method stores its return value the first time it runs; subsequent calls with the same arguments return the cached value without executing the method body. @CachePut always runs the method and updates the cache with the result — useful for update operations. @CacheEvict removes an entry (or all entries) from the cache — used on delete operations. Spring supports multiple cache providers: the default in-memory ConcurrentMapCache, Caffeine, Redis, EHCache, and others.

Java 
@SpringBootApplication
@EnableCaching              // activate the caching infrastructure
public class Application { }

@Service
public class ProductService {

    // @Cacheable — cache result; skip method on cache hit
    @Cacheable(value = "products", key = "#id")
    public Product findById(Long id) {
        System.out.println("DB hit for id: " + id); // only printed on first call
        return productRepo.findById(id).orElseThrow();
    }

    // @CachePut — always run + update cache entry
    @CachePut(value = "products", key = "#product.id")
    public Product update(Product product) {
        return productRepo.save(product);  // cache refreshed with new value
    }

    // @CacheEvict — remove from cache on delete
    @CacheEvict(value = "products", key = "#id")
    public void delete(Long id) {
        productRepo.deleteById(id);
    }

    // @CacheEvict with allEntries=true — clear entire cache
    @CacheEvict(value = "products", allEntries = true)
    public void clearAll() { }
}
Q25

What is Spring's @Async annotation? How does it enable asynchronous method execution?

Data ▼

@Async lets you run a method on a separate thread from a thread pool without manually managing threads. Add @EnableAsync to your config and @Async to any method. When the method is called, Spring intercepts the call (via AOP), submits the task to an executor thread pool, and immediately returns to the caller — the method body runs asynchronously in the background. Methods can return void (fire-and-forget) or CompletableFuture<T> (so the caller can retrieve the result later or compose async operations). Important: @Async only works on Spring beans called from OUTSIDE the bean (proxy limitation) — a method calling this.asyncMethod() internally will NOT be async.

Java 
@SpringBootApplication
@EnableAsync
public class Application { }

@Service
public class NotificationService {

    // Fire-and-forget — caller continues immediately, email sent in background
    @Async
    public void sendWelcomeEmail(String email) {
        Thread.sleep(2000);    // simulate slow email send
        System.out.println("Email sent to: " + email);
    }

    // Returns result later via CompletableFuture
    @Async
    public CompletableFuture<String> processReport(Long reportId) {
        String result = heavyReportGeneration(reportId);
        return CompletableFuture.completedFuture(result);
    }
}

// Custom thread pool instead of the default SimpleAsyncTaskExecutor
@Bean
public Executor taskExecutor() {
    ThreadPoolTaskExecutor exec = new ThreadPoolTaskExecutor();
    exec.setCorePoolSize(5);
    exec.setMaxPoolSize(20);
    exec.setQueueCapacity(100);
    exec.setThreadNamePrefix("async-");
    exec.initialize();
    return exec;
}
Q26

What is Spring's @Scheduled annotation? How do you set up scheduled tasks?

Data ▼

@Scheduled turns a method into a background scheduled task — no external scheduler or cron daemon needed. Add @EnableScheduling to your configuration. Tasks can be triggered in three ways: fixedRate runs the method every N milliseconds from the previous start time — useful for polling. fixedDelay waits N milliseconds after the previous execution COMPLETES — useful for sequential tasks that must not overlap. cron uses a cron expression for calendar-based scheduling (every day at 2am, every Monday at 9am). By default, all scheduled tasks share a single thread — long tasks block others. Configure a TaskScheduler bean with a thread pool to run them in parallel.

Java 
@SpringBootApplication
@EnableScheduling
public class Application { }

@Component
public class ScheduledTasks {

    // Run every 5 seconds from the start of previous run
    @Scheduled(fixedRate = 5000)
    public void pollExternalApi() {
        System.out.println("Polling at: " + LocalTime.now());
    }

    // Wait 10 seconds AFTER previous run completes before running again
    @Scheduled(fixedDelay = 10000)
    public void processQueue() {
        System.out.println("Processing queue...");
    }

    // Cron: runs at 2:30 AM every day
    @Scheduled(cron = "0 30 2 * * *")
    public void dailyCleanup() {
        System.out.println("Running daily cleanup...");
    }

    // Cron: every Monday at 9:00 AM
    @Scheduled(cron = "0 0 9 * * MON")
    public void weeklyReport() {
        System.out.println("Generating weekly report...");
    }

    // Cron expression from properties file (flexible, no recompile)
    @Scheduled(cron = "${app.cleanup.cron:0 0 3 * * *}")
    public void configurableTask() { }
}
Q27

What is @SpringBootTest? When should you use it and what are its modes?

Testing ▼

@SpringBootTest loads the complete Spring application context for integration testing — it is as close as you can get to running the real app in a test. It finds your @SpringBootApplication class and boots the full context with all beans, auto configuration, and properties. The webEnvironment attribute controls the server: MOCK (default) — loads a mock servlet environment without starting a real HTTP server (use with MockMvc). RANDOM_PORT — starts a real embedded Tomcat on a random port (use with TestRestTemplate or WebTestClient). NONE — no web environment at all. Use @SpringBootTest for integration tests. For unit tests, prefer plain JUnit + Mockito — loading the full context is slow and unnecessary.

Java 
// Integration test with mock web environment (no real HTTP server)
@SpringBootTest
@AutoConfigureMockMvc
class ProductControllerIntegrationTest {

    @Autowired
    private MockMvc mockMvc;

    @Test
    void getProduct_returnsOk() throws Exception {
        mockMvc.perform(get("/api/products/1"))
               .andExpect(status().isOk())
               .andExpect(jsonPath("$.name").value("Widget"));
    }
}

// Integration test with REAL embedded server on a random port
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
class FullStackTest {

    @Autowired
    private TestRestTemplate restTemplate;

    @LocalServerPort         // injects the random port chosen
    private int port;

    @Test
    void healthCheck() {
        ResponseEntity<String> resp =
            restTemplate.getForEntity("http://localhost:" + port + "/actuator/health", String.class);
        Assertions.assertEquals(HttpStatus.OK, resp.getStatusCode());
    }
}
Q28

What is @MockBean and how is it different from Mockito's @Mock?

Testing ▼

@Mock (from Mockito) creates a mock object in plain unit tests — it has no awareness of Spring and the mock is only available in the test class via @ExtendWith(MockitoExtension.class). @MockBean (from Spring Boot Test) creates a Mockito mock AND registers it as a Spring bean in the application context, replacing the real bean. This means all Spring beans that depend on it will receive the mock. Use @Mock in pure unit tests where you instantiate the class under test directly. Use @MockBean in slice tests (@WebMvcTest) or @SpringBootTest where you need to replace a real Spring-managed dependency with a mock.

Java 
// Pure unit test with @Mock — no Spring context needed
@ExtendWith(MockitoExtension.class)
class OrderServiceTest {

    @Mock
    private OrderRepository orderRepository;   // Mockito mock, not a Spring bean

    @InjectMocks
    private OrderService orderService;          // manually wired by Mockito

    @Test
    void placeOrder_success() {
        when(orderRepository.save(any())).thenReturn(new Order(1L));
        Order result = orderService.place(new OrderRequest());
        Assertions.assertNotNull(result);
    }
}

// Slice test with @MockBean — Spring loads only the web layer
@WebMvcTest(OrderController.class)
class OrderControllerTest {

    @Autowired
    private MockMvc mockMvc;

    @MockBean
    private OrderService orderService;   // replaces real bean in Spring context

    @Test
    void getOrder_returnsMockedData() throws Exception {
        when(orderService.findById(1L)).thenReturn(Optional.of(new Order(1L)));
        mockMvc.perform(get("/api/orders/1"))
               .andExpect(status().isOk());
    }
}
Q29

What is @WebMvcTest? How is it used for testing controllers in isolation?

Testing ▼

@WebMvcTest is a slice test annotation that loads only the web layer of your application — controllers, filters, @ControllerAdvice, Jackson configuration, and MockMvc. It does NOT load the service or repository layer, making tests much faster than a full @SpringBootTest. You mock out all service dependencies with @MockBean. This lets you test controller logic, request mapping, request/ response serialisation, validation, HTTP status codes, and exception handling in pure isolation. Pass the specific controller class to the annotation to load only that controller and avoid ambiguity when multiple controllers exist.

Java 
@WebMvcTest(ProductController.class)      // only loads ProductController
class ProductControllerTest {

    @Autowired
    private MockMvc mockMvc;                 // auto-configured

    @Autowired
    private ObjectMapper objectMapper;       // for JSON serialisation

    @MockBean
    private ProductService productService;  // service is mocked, not real

    @Test
    void getProduct_found_returns200() throws Exception {
        Product p = new Product(1L, "Widget", 9.99);
        when(productService.findById(1L)).thenReturn(Optional.of(p));

        mockMvc.perform(get("/api/products/1")
                   .contentType(MediaType.APPLICATION_JSON))
               .andExpect(status().isOk())
               .andExpect(jsonPath("$.name").value("Widget"))
               .andExpect(jsonPath("$.price").value(9.99));
    }

    @Test
    void createProduct_invalidBody_returns400() throws Exception {
        Product invalid = new Product(null, "", -1);  // fails @NotBlank, @Positive

        mockMvc.perform(post("/api/products")
                   .contentType(MediaType.APPLICATION_JSON)
                   .content(objectMapper.writeValueAsString(invalid)))
               .andExpect(status().isBadRequest());
    }
}
Q30

What is the difference between unit testing and integration testing in Spring Boot? How do you decide which to write?

Testing ▼

A unit test tests a single class in complete isolation — all dependencies are mocked (using Mockito), no Spring context is loaded, and the test runs in milliseconds. It verifies the logic of one class: does the service compute the right result? Does it call the repository with the right arguments? An integration test tests how multiple components work together — the controller + service + repository against a real (or in-memory) database. It loads a Spring context (with @SpringBootTest or a slice annotation) and is slower. A practical strategy: write unit tests for all business logic (fast feedback loop). Write integration tests for the critical paths and API contract (ensure layers wire together correctly). Use the test pyramid: many unit tests, fewer integration tests, very few end-to-end tests.

Java 
// ── UNIT TEST: no Spring, no DB, pure Java + Mockito ──────────────────
@ExtendWith(MockitoExtension.class)
class DiscountServiceTest {

    @Mock  UserRepository userRepo;
    @InjectMocks  DiscountService discountService;

    @Test
    void premiumUser_gets20PercentDiscount() {
        when(userRepo.findById(1L)).thenReturn(Optional.of(new User(1L, true)));
        double discount = discountService.calculate(1L, 100.0);
        Assertions.assertEquals(20.0, discount);
    }
}

// ── INTEGRATION TEST: real Spring context + in-memory H2 DB ───────────
@SpringBootTest
@Transactional          // rolls back DB changes after each test
class OrderFlowIntegrationTest {

    @Autowired  OrderService    orderService;
    @Autowired  OrderRepository orderRepo;

    @Test
    void placeOrder_savedToDatabase() {
        OrderRequest req = new OrderRequest("Alice", 3);
        Order placed = orderService.place(req);

        Order found = orderRepo.findById(placed.getId()).orElseThrow();
        Assertions.assertEquals("Alice", found.getCustomerName());
    }
}
