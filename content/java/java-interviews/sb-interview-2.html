Spring Boot questions on JPA

Q1. What is JPA and how is it different from JDBC and Hibernate?

JPA (Java Persistence API) is a specification — a set of interfaces and rules defined by Jakarta EE that describes how Java objects should be mapped to relational database tables and how to query them. It does not provide any implementation itself. JDBC is a low-level API where you write raw SQL, manage connections manually, and map ResultSet rows to objects by hand — it gives full control but produces a lot of boilerplate. Hibernate is the most popular implementation of JPA — it does the heavy lifting of translating your annotated Java classes into SQL queries. Spring Boot auto-configures Hibernate as the JPA provider when you add spring-boot-starter-data-jpa. Think of it this way: JPA is the contract, Hibernate is the implementation, and JDBC is the low-level foundation they all sit on top of.
Java 

    // JDBC — raw, verbose, manual mapping
    Connection conn = DriverManager.getConnection(url, user, pass);
    PreparedStatement ps = conn.prepareStatement("SELECT * FROM users WHERE id = ?");
    ps.setLong(1, id);
    ResultSet rs = ps.executeQuery();
    if (rs.next()) {
        User u = new User();
        u.setId(rs.getLong("id"));
        u.setName(rs.getString("name"));   // manual mapping — tedious
    }

    // JPA — annotate your class, no SQL needed for basic operations
    @Entity
    @Table(name = "users")
    public class User {
        @Id
        @GeneratedValue(strategy = GenerationType.IDENTITY)
        private Long id;

        private String name;
        // getters and setters
    }

    // Spring Data JPA — find by ID with zero SQL
    User user = userRepository.findById(id).orElseThrow();

Q2. What is an Entity in JPA? What are the rules a class must follow to be an entity?

An entity is a Java class that is mapped to a database table. Each instance of the entity represents one row in that table. To be a valid JPA entity, the class must follow certain rules: it must be annotated with @Entity, it must have a no-argument constructor (can be protected or public — JPA needs it to create instances via reflection), it must not be final (because JPA providers like Hibernate create proxy subclasses for lazy loading), and it must have a field annotated with @Id that represents the primary key. Fields are mapped to columns by default using the field name. You can customise the table name with @Table and column names with @Column.
Java 

@Entity                             // marks this class as a JPA entity
@Table(name = "products")           // maps to the 'products' table (optional if name matches)
public class Product {

    @Id                             // primary key
    @GeneratedValue(strategy = GenerationType.IDENTITY)  // auto-increment
    private Long id;

    @Column(name = "product_name", nullable = false, length = 100)
    private String name;

    @Column(precision = 10, scale = 2)
    private BigDecimal price;

    @Column(name = "created_at", updatable = false)
    private LocalDateTime createdAt;

    // Required: no-arg constructor (JPA uses it via reflection)
    public Product() {}

    public Product(String name, BigDecimal price) {
        this.name  = name;
        this.price = price;
    }
    // getters and setters
}

Q3. What is the difference between @Id and @GeneratedValue? What generation strategies are available?

@Id marks the field as the primary key of the entity. @GeneratedValue tells JPA how to automatically generate the primary key value — without it, you must set the ID manually before saving. There are four strategies. IDENTITY delegates key generation to the database's auto-increment column (most common with MySQL, PostgreSQL). SEQUENCE uses a database sequence object — the most efficient strategy for Hibernate because it can batch ID allocation (preferred with PostgreSQL, Oracle). TABLE uses a separate database table to simulate a sequence — portable but slow, rarely used. AUTO lets JPA pick the best strategy based on the database dialect. IDENTITY and SEQUENCE are the two you will use in almost every real project.
Java 

// IDENTITY — relies on database auto-increment (MySQL, PostgreSQL SERIAL)
@Id
@GeneratedValue(strategy = GenerationType.IDENTITY)
private Long id;

// SEQUENCE — uses a DB sequence, allows Hibernate to batch allocate IDs
@Id
@GeneratedValue(strategy = GenerationType.SEQUENCE, generator = "product_seq")
@SequenceGenerator(name = "product_seq", sequenceName = "product_sequence",
                   allocationSize = 50)  // fetch 50 IDs at a time for performance
private Long id;

// AUTO — JPA picks based on database (default if strategy omitted)
@Id
@GeneratedValue(strategy = GenerationType.AUTO)
private Long id;

// Manual ID — no @GeneratedValue, you set it yourself
@Id
private String isoCode;   // e.g., "USD", "EUR" — natural keys

// UUID primary key (common in distributed systems)
@Id
@GeneratedValue(strategy = GenerationType.UUID)   // JPA 3.1 / Hibernate 6+
private UUID id;

Q4. What is Spring Data JPA? How does JpaRepository work and what does it provide out of the box?

Spring Data JPA sits on top of JPA and eliminates even more boilerplate by letting you define repository interfaces with zero implementation code. You extend JpaRepository<EntityType, IdType> and Spring generates the implementation at runtime using dynamic proxies. Out of the box you get: save() and saveAll() for insert and update, findById(), findAll(), findAllById(), count(), existsById(), delete(), deleteById(), deleteAll(), and flush(). It also provides PagingAndSortingRepository methods for pagination and sorting. You do not write a single line of SQL or JPQL for these common operations. You can also add custom query methods just by naming them in a specific way — Spring parses the method name and generates the query automatically.
Java 

// Define the interface — Spring generates the implementation at runtime
public interface ProductRepository extends JpaRepository {
    // No code needed for basic CRUD
}

// Using the repository in a service
@Service
@Transactional
public class ProductService {

    private final ProductRepository productRepository;

    public ProductService(ProductRepository repo) { this.productRepository = repo; }

    public Product create(Product p)          { return productRepository.save(p); }
    public Optional find(Long id)    { return productRepository.findById(id); }
    public List findAll()            { return productRepository.findAll(); }
    public long count()                       { return productRepository.count(); }
    public void delete(Long id)               { productRepository.deleteById(id); }
    public boolean exists(Long id)            { return productRepository.existsById(id); }

    // Pagination — findAll with a Pageable argument
    public Page findPaged(int page, int size) {
        return productRepository.findAll(PageRequest.of(page, size, Sort.by("name")));
    }

    // Save updates too — JPA merges if ID already exists
    public Product update(Product p)          { return productRepository.save(p); }
}

Q5. What are derived query methods in Spring Data JPA? How does Spring know what query to generate?

Derived query methods let you define queries simply by writing a method name that follows a naming convention — Spring Data JPA parses the name and generates the corresponding JPQL query at application startup. The method name starts with a prefix like findBy, countBy, existsBy, or deleteBy, followed by property names of your entity joined with keywords like And, Or, Between, LessThan, GreaterThan, Like, Containing, StartingWith, OrderBy, Not, In, and IsNull. Spring inspects your entity's fields, validates that the property names exist, and generates the query. If a property does not exist, the application fails to start — not at query execution time — which gives you early error detection.
Java 

public interface ProductRepository extends JpaRepository {

    // SELECT * FROM products WHERE name = ?
    List findByName(String name);

    // SELECT * FROM products WHERE price < ? AND category = ?
    List findByPriceLessThanAndCategory(BigDecimal price, String category);

    // SELECT * FROM products WHERE name LIKE %?%
    List findByNameContainingIgnoreCase(String keyword);

    // SELECT * FROM products WHERE price BETWEEN ? AND ?
    List findByPriceBetween(BigDecimal min, BigDecimal max);

    // SELECT * FROM products WHERE active = true ORDER BY name ASC
    List findByActiveTrueOrderByNameAsc();

    // SELECT * FROM products WHERE category IN (?, ?, ?)
    List findByCategoryIn(List categories);

    // SELECT COUNT(*) FROM products WHERE category = ?
    long countByCategory(String category);

    // SELECT * FROM products WHERE name IS NOT NULL
    List findByNameIsNotNull();

    // Returns only one result — throws if multiple match
    Optional findBySkuCode(String skuCode);
}

Q6. What is JPQL and how is it different from native SQL? When would you use @Query?

JPQL (Java Persistence Query Language) is an object-oriented query language that operates on entity classes and their fields, not on database tables and columns. It is database-agnostic — JPA translates JPQL into the appropriate SQL dialect for your database at runtime. For example, in JPQL you write FROM Product p WHERE p.name = :name (using the class name Product, not the table name products). Native SQL queries use the actual table and column names and are database-specific. Use @Query when your logic is too complex for a derived method name — JOINs, aggregations, subqueries, or projections. Use nativeQuery = true when you need database-specific SQL features that JPQL cannot express.
Java 

public interface ProductRepository extends JpaRepository {

    // JPQL — uses entity class name 'Product' and field name 'price', not table/column
    @Query("SELECT p FROM Product p WHERE p.price > :minPrice ORDER BY p.price DESC")
    List findExpensive(@Param("minPrice") BigDecimal minPrice);

    // JPQL with JOIN
    @Query("SELECT p FROM Product p JOIN p.category c WHERE c.name = :catName")
    List findByCategory(@Param("catName") String catName);

    // JPQL with aggregation
    @Query("SELECT AVG(p.price) FROM Product p WHERE p.category.name = :cat")
    Double avgPriceByCategory(@Param("cat") String cat);

    // Native SQL — uses actual table and column names, database-specific
    @Query(value = "SELECT * FROM products WHERE YEAR(created_at) = :year",
           nativeQuery = true)
    List findCreatedInYear(@Param("year") int year);

    // Modifying query — UPDATE or DELETE, must use @Modifying
    @Modifying
    @Transactional
    @Query("UPDATE Product p SET p.active = false WHERE p.price < :minPrice")
    int deactivateCheapProducts(@Param("minPrice") BigDecimal minPrice);
}

Q7. What are the JPA entity states — Transient, Managed, Detached, and Removed?

Every JPA entity instance lives in one of four states that determine how the EntityManager tracks it. A Transient entity is a newly created Java object with no ID — JPA has no awareness of it and it is not in the database. A Managed (or Persistent) entity has been associated with an active EntityManager (persistence context) — any changes you make to it are automatically detected and synced to the database when the transaction commits (this is called dirty checking). A Detached entity was once managed but its persistence context has closed (after the transaction ends, or after a call to detach()) — changes to it are NOT tracked. A Removed entity is scheduled for deletion — JPA will delete its row when the transaction commits.
Java 

@Service
@Transactional
public class UserService {

    @PersistenceContext
    private EntityManager em;

    public void demonstrateStates() {

        // 1. TRANSIENT — new object, not known to JPA, no DB row
        User user = new User("Alice");
        // user has no id, EntityManager does not track it

        // 2. MANAGED — after persist(), EntityManager tracks it
        em.persist(user);
        // user is now managed: any field change is auto-saved on commit
        user.setName("Alice Smith");   // no explicit save() needed!
        // transaction commits → UPDATE users SET name='Alice Smith' WHERE id=?

        // 3. DETACHED — after context closes or explicit detach
        em.detach(user);
        user.setName("Alice Jones");  // change NOT tracked, NOT saved to DB

        // Re-attach a detached entity
        User managed = em.merge(user);  // managed copy returned, changes applied

        // 4. REMOVED — scheduled for deletion
        em.remove(managed);
        // transaction commits → DELETE FROM users WHERE id=?
    }
}

Q8. What is the Persistence Context and what is the difference between EntityManager and Session?

The persistence context is a first-level cache managed by the EntityManager — it acts as an identity map between entity IDs and managed entity instances. Within a single persistence context (typically one transaction), if you load the same entity twice by ID, JPA returns the same Java object reference without hitting the database a second time. The EntityManager is the JPA standard interface for interacting with the persistence context — it provides persist(), merge(), remove(), find(), and createQuery(). Session is Hibernate's own equivalent interface — it extends the JPA EntityManager but adds Hibernate-specific methods. In Spring Boot with JPA, you always code against EntityManager; the underlying Session is available if you need Hibernate-specific features via entityManager.unwrap(Session.class).
Java 

@Service
@Transactional
public class DemoService {

    @PersistenceContext     // Spring injects a thread-safe proxy
    private EntityManager em;

    public void persistenceContextDemo() {

        // First load — hits the database
        User u1 = em.find(User.class, 1L);

        // Second load — returns SAME instance from first-level cache, no DB hit
        User u2 = em.find(User.class, 1L);

        System.out.println(u1 == u2);   // true — same Java object reference

        // Dirty checking — no save() call needed
        u1.setEmail("new@example.com");
        // on commit: Hibernate compares snapshot vs current state → issues UPDATE
    }

    public void useHibernateSpecificFeatures() {
        // Unwrap to Hibernate Session for Hibernate-specific API
        Session session = em.unwrap(Session.class);

        // Hibernate-specific: load a proxy without hitting DB
        User proxy = session.getReference(User.class, 1L);

        // Hibernate multi-load (batch load by IDs)
        List users = session.byMultipleIds(User.class)
                                  .multiLoad(1L, 2L, 3L);
    }
}

Q9. What is the difference between save(), saveAndFlush(), and flush() in Spring Data JPA?

In Spring Data JPA, save() calls persist() for new entities (no ID) or merge() for existing ones (has an ID), and it queues the SQL statement — the actual INSERT or UPDATE is sent to the database when Hibernate decides to flush, which normally happens at the end of the transaction. saveAndFlush() does the same as save() but immediately forces Hibernate to synchronise the persistence context with the database (sends the SQL right now) within the same transaction. flush() alone forces synchronisation without returning the entity. Use saveAndFlush() when you need the database to reflect your changes immediately within the same transaction — for example, before executing a native query that would otherwise not see the pending changes.
Java 

@Service
@Transactional
public class OrderService {

    private final OrderRepository orderRepository;

    public void demonstrateSave() {

        Order order = new Order("ORD-001", BigDecimal.valueOf(99.99));

        // save() — persists/merges but SQL may not be sent to DB yet
        Order saved = orderRepository.save(order);
        // SQL is queued; DB may not have the row yet during this transaction

        // saveAndFlush() — forces immediate SQL execution to the DB
        Order flushed = orderRepository.saveAndFlush(order);
        // INSERT INTO orders ... is sent to DB RIGHT NOW
        // (still within transaction — can still be rolled back)

        // Use case: native query won't see changes unless flushed first
        Order o = new Order("ORD-002", BigDecimal.valueOf(50.0));
        orderRepository.saveAndFlush(o);   // ensure DB row exists
        // Now this native query can see it:
        orderRepository.findByNativeQuery("ORD-002");

        // flush() alone — synchronise without returning entity
        orderRepository.flush();
    }
}

Q10. What is lazy loading vs eager loading in JPA? What is the N+1 problem and how do you solve it?

FetchType.LAZY means JPA loads the related data only when you actually access it — the association is a proxy that fires a SQL query on first touch. FetchType.EAGER means JPA loads the related data immediately in the same query when the parent is loaded. LAZY is the default for @OneToMany and @ManyToMany; EAGER is the default for @ManyToOne and @OneToOne. The N+1 problem is the most common JPA performance issue: if you load N orders and each order lazily loads its customer, you fire 1 query to get all orders, then N separate queries to load each customer — N+1 total database hits. The solution is to use JOIN FETCH in JPQL or @EntityGraph to load the association in a single JOIN query.
Java 

// N+1 PROBLEM — 1 query for orders + N queries for each customer
List orders = orderRepository.findAll();   // 1 query
for (Order o : orders) {
    System.out.println(o.getCustomer().getName());// N queries (lazy load each)
}
// If there are 100 orders → 101 database queries!

// SOLUTION 1: JOIN FETCH in JPQL — single query with JOIN
@Query("SELECT o FROM Order o JOIN FETCH o.customer")
List findAllWithCustomer();

// SOLUTION 2: @EntityGraph — load association without writing JPQL
@EntityGraph(attributePaths = {"customer", "items"})
@Query("SELECT o FROM Order o")
List findAllWithCustomerAndItems();

// SOLUTION 3: @NamedEntityGraph on the entity
@Entity
@NamedEntityGraph(name = "Order.withCustomer",
    attributeNodes = @NamedAttributeNode("customer"))
public class Order {
    @ManyToOne(fetch = FetchType.LAZY)   // LAZY is the right default
    private Customer customer;
}

// Repository using the named graph
@EntityGraph("Order.withCustomer")
List findByStatus(String status);

Q11. What are the JPA relationship annotations? Explain @OneToOne, @OneToMany, @ManyToOne, and @ManyToMany.

JPA provides four annotations to model database relationships between entities. @ManyToOne is the most common — many orders belong to one customer. It maps the foreign key column and is EAGER by default. @OneToMany is the inverse — one customer has many orders. It is LAZY by default and is usually the mappedBy side. @OneToOne maps a one-to-one relationship — one user has one profile. @ManyToMany maps a many-to-many relationship — students and courses — JPA manages a join table automatically. Always define the owning side (the one with the foreign key or @JoinColumn) and the inverse side (with mappedBy pointing back to the owning field). The mappedBy side is read-only for the relationship — you must manage the owning side to make changes.
Java 

// @ManyToOne — owning side, has the foreign key column
@Entity
public class Order {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @ManyToOne(fetch = FetchType.LAZY)  // override EAGER default
    @JoinColumn(name = "customer_id")   // FK column in orders table
    private Customer customer;
}

// @OneToMany — inverse side, no FK column here
@Entity
public class Customer {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @OneToMany(mappedBy = "customer", cascade = CascadeType.ALL,
               orphanRemoval = true, fetch = FetchType.LAZY)
    private List orders = new ArrayList<>();

    // Helper method to keep both sides in sync
    public void addOrder(Order order) {
        orders.add(order);
        order.setCustomer(this);   // always set the owning side
    }
}

// @ManyToMany — JPA creates a join table automatically
@Entity
public class Student {
    @ManyToMany
    @JoinTable(name = "student_course",
        joinColumns        = @JoinColumn(name = "student_id"),
        inverseJoinColumns = @JoinColumn(name = "course_id"))
    private Set courses = new HashSet<>();
}

@Entity
public class Course {
    @ManyToMany(mappedBy = "courses")   // inverse side
    private Set students = new HashSet<>();
}

Q12. What is CascadeType in JPA? What are the different cascade types and when do you use them?

Cascade means that a JPA operation performed on a parent entity is automatically applied to its related child entities. Without cascade, you must explicitly persist, merge, or remove each child manually. There are six cascade types. PERSIST — saving the parent auto-saves children. MERGE — merging the parent auto-merges children. REMOVE — deleting the parent auto-deletes children. REFRESH — refreshing the parent refreshes children from DB. DETACH — detaching the parent detaches children. ALL — applies all of the above. CascadeType.ALL with orphanRemoval = true is the correct combination for a parent-child (aggregate) relationship where children have no meaning without the parent — deleting an Order should delete its OrderItems.
Java 

@Entity
public class Order {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    // CascadeType.ALL + orphanRemoval — fully managed child lifecycle
    @OneToMany(mappedBy = "order",
               cascade = CascadeType.ALL,    // persist/merge/remove cascade
               orphanRemoval = true,          // delete item if removed from list
               fetch = FetchType.LAZY)
    private List items = new ArrayList<>();

    public void addItem(OrderItem item) {
        items.add(item);
        item.setOrder(this);
    }

    public void removeItem(OrderItem item) {
        items.remove(item);   // orphanRemoval=true → item deleted from DB
        item.setOrder(null);
    }
}

// Service — cascade means you only save/delete the parent
@Service
@Transactional
public class OrderService {

    public Order create(Order order) {
        // Saving order auto-persists all OrderItems (CascadeType.PERSIST)
        return orderRepository.save(order);
    }

    public void delete(Long orderId) {
        // Deleting order auto-deletes all OrderItems (CascadeType.REMOVE)
        orderRepository.deleteById(orderId);
    }
}

// CascadeType.PERSIST only — share a Category across Products (no cascade delete)
@ManyToOne(cascade = CascadeType.PERSIST, fetch = FetchType.LAZY)
@JoinColumn(name = "category_id")
private Category category;

Q13. What is the difference between @Transient and transient in JPA?

Both tell JPA to skip a field when mapping to the database, but they work differently. The @Transient annotation (from javax.persistence / jakarta.persistence) is the JPA-specific way to exclude a field — the field is not persisted to or read from the database, but it is still available for serialization (e.g., to JSON via Jackson). The Java transient keyword is a core Java keyword that excludes a field from Java object serialization (ObjectOutputStream) but JPA also honours it — the field will not be persisted. In practice, always use @Transient for JPA entities because it is clear in intent, it works with all JPA providers consistently, and it does not accidentally break your JSON serialization.
Java 

@Entity
public class Product {

    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;
    private BigDecimal price;

    // @Transient — NOT persisted to DB, but IS included in JSON serialization
    @Transient
    private BigDecimal discountedPrice;    // computed at runtime, not stored

    // Java 'transient' keyword — NOT persisted to DB, also NOT in Java serialization
    private transient int cachedHashCode;  // local cache, meaningless to persist

    // A common use: computed/derived field
    @PostLoad                              // called by JPA after loading from DB
    public void calculateDiscount() {
        this.discountedPrice = price.multiply(BigDecimal.valueOf(0.9)); // 10% off
    }

    // @Transient field appears in JSON response (Jackson sees it)
    // transient field does NOT appear in JSON (Jackson respects Java transient)
}

Q14. What is the @Embedded and @Embeddable annotation in JPA? When would you use them?

@Embeddable marks a class whose fields should be stored in the same database table as the owning entity — it has no table of its own and no @Id. @Embedded is used in the entity to include the embeddable object. This is the JPA way to implement value objects — objects that are defined by their fields rather than an identity. A classic example is Address: instead of adding street, city, zip, country directly to User and Order (duplicating the columns), you define an Address embeddable and embed it in both. The address fields are stored in the user and order tables respectively. This keeps your domain model clean and object-oriented while the database stays flat.
Java 

// Value object — @Embeddable, no @Id, no @Entity
@Embeddable
public class Address {
    @Column(name = "street", nullable = false)
    private String street;

    @Column(name = "city", nullable = false)
    private String city;

    @Column(name = "zip_code", length = 10)
    private String zipCode;

    @Column(name = "country", length = 2)
    private String country;

    protected Address() {}

    public Address(String street, String city, String zipCode, String country) {
        this.street  = street;
        this.city    = city;
        this.zipCode = zipCode;
        this.country = country;
    }
}

// Customer entity — Address fields stored in the 'customers' table
@Entity
public class Customer {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    private String name;

    @Embedded                          // address fields added to this table
    private Address shippingAddress;

    // Override column names when embedding the same type twice
    @Embedded
    @AttributeOverrides({
        @AttributeOverride(name = "street",  column = @Column(name = "billing_street")),
        @AttributeOverride(name = "city",    column = @Column(name = "billing_city")),
        @AttributeOverride(name = "zipCode", column = @Column(name = "billing_zip"))
    })
    private Address billingAddress;
}

Q15. What is optimistic locking in JPA? How does @Version work?

Optimistic locking is a concurrency control strategy based on the assumption that conflicts are rare. Rather than locking the database row (which hurts performance), each entity tracks a version number. When two users load the same entity and both try to update it, JPA includes the version number in the WHERE clause of the UPDATE. The first user's update succeeds and increments the version. When the second user tries to update, JPA generates UPDATE ... WHERE id = ? AND version = 1 — but the row now has version = 2, so zero rows are updated, and JPA throws OptimisticLockException. You add optimistic locking by adding a field annotated with @Version — JPA manages the version field entirely. The column can be an int, long, Integer, Long, or Timestamp.
Java 

@Entity
public class BankAccount {

    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private BigDecimal balance;

    @Version                       // JPA manages this field automatically
    private Long version;          // starts at 0, incremented on every update

    // Getters and setters
}

// What JPA generates under the hood on update:
// UPDATE bank_account
//    SET balance = 900.00, version = 2
//  WHERE id = 1 AND version = 1    ← version check prevents lost update
//
// If another thread already updated version to 2:
//   → 0 rows updated → JPA throws OptimisticLockException

@Service
@Transactional
public class BankService {

    public void withdraw(Long accountId, BigDecimal amount) {
        try {
            BankAccount account = accountRepository.findById(accountId).orElseThrow();
            account.setBalance(account.getBalance().subtract(amount));
            // save() triggers the version-checked UPDATE
        } catch (OptimisticLockException e) {
            // Another transaction modified this account — retry or notify user
            throw new ConcurrentUpdateException("Account was modified. Please retry.");
        }
    }
}

Q16. What is the difference between find(), getReference(), and a JPQL query in JPA?

find() immediately executes a SELECT query and returns the fully initialised entity, or null if not found. getReference() returns a Hibernate proxy object without hitting the database — the SELECT is deferred until you access a field other than the ID. This is useful for setting up relationships without an unnecessary load (you only need the ID to set a foreign key). A JPQL query via createQuery() or @Query gives you full control over what you select, filter, join, and project — essential for anything beyond simple ID lookups. Knowing when to use each matters for performance: use find() when you need the full entity, getReference() when you only need to establish a relationship, and JPQL/derived queries when you query by anything other than the primary key.
Java 

@Service
@Transactional
public class OrderService {

    @PersistenceContext
    private EntityManager em;

    private final CategoryRepository categoryRepository;

    public void demonstrate(Long orderId, Long categoryId) {

        // find() — immediate SELECT, returns null if not found
        Order order = em.find(Order.class, orderId);
        if (order == null) throw new EntityNotFoundException("Order not found");

        // getReference() — returns a PROXY, no SELECT until field accessed
        // Use case: set a foreign key without loading the full Category
        Category categoryProxy = em.getReference(Category.class, categoryId);
        order.setCategory(categoryProxy);
        // Only the FK column is set; no SELECT on Category table

        // Spring Data equivalent of getReference()
        Category ref = categoryRepository.getReferenceById(categoryId);

        // JPQL — flexible, full control
        List recent = em.createQuery(
            "SELECT o FROM Order o WHERE o.createdAt > :since ORDER BY o.createdAt DESC",
            Order.class)
            .setParameter("since", LocalDateTime.now().minusDays(7))
            .setMaxResults(20)
            .getResultList();
    }
}

Q17. What is JPQL projection and DTO projection? Why would you project instead of loading full entities?

Loading full entities is wasteful when you only need a few fields — Hibernate loads every column, creates managed entity proxies, and tracks changes for dirty checking, all for data you never modify. A projection loads only the columns you need. There are three approaches. Interface-based projection: define an interface with getters matching field names — Spring Data creates a proxy that returns only those columns. Class-based projection (DTO): define a record or class with a constructor matching the query — Spring passes values directly. @Query with JPQL constructor expression: use the new keyword in JPQL to map results to a DTO. Projections reduce network traffic, reduce memory usage, skip dirty checking, and often produce simpler SQL.
Java 

// 1. Interface projection — Spring generates proxy at runtime
public interface ProductSummary {
    Long   getId();
    String getName();
    BigDecimal getPrice();   // only 3 columns fetched, not all
}

public interface ProductRepository extends JpaRepository {
    List findByCategory(String category);
}

// 2. Class (DTO/Record) projection — direct mapping via constructor
public record ProductDTO(Long id, String name, BigDecimal price) {}

public interface ProductRepository extends JpaRepository {
    @Query("SELECT new com.example.dto.ProductDTO(p.id, p.name, p.price) " +
           "FROM Product p WHERE p.category = :cat")
    List findSummaryByCategory(@Param("cat") String cat);
}

// 3. Dynamic projection — return type determined by the caller
public interface ProductRepository extends JpaRepository {
     List findByActive(boolean active, Class type);
}

// Usage:
List summaries = productRepository.findByActive(true, ProductSummary.class);
List        entities  = productRepository.findByActive(true, Product.class);

Q18. What are @PrePersist, @PostPersist, @PreUpdate, and other JPA lifecycle callbacks?

JPA provides lifecycle callback annotations that let you execute logic at specific points in an entity's lifecycle automatically — without any explicit calls in your service code. @PrePersist runs just before the entity is first inserted into the database — ideal for setting createdAt timestamps or generating codes. @PostPersist runs just after the insert. @PreUpdate runs before an UPDATE — ideal for setting updatedAt. @PostLoad runs after the entity is loaded from the database. @PreRemove runs before deletion. These callbacks are defined as methods inside the entity class itself. For reusable audit logic across many entities, use a @MappedSuperclass containing the callback methods and have your entities extend it.
Java 

// Reusable audit base class — extend this in any entity you want audited
@MappedSuperclass
public abstract class Auditable {

    @Column(name = "created_at", nullable = false, updatable = false)
    private LocalDateTime createdAt;

    @Column(name = "updated_at")
    private LocalDateTime updatedAt;

    @PrePersist
    protected void onCreate() {
        createdAt = LocalDateTime.now();
        updatedAt = LocalDateTime.now();
    }

    @PreUpdate
    protected void onUpdate() {
        updatedAt = LocalDateTime.now();
    }
}

// Entity — gets audit fields for free
@Entity
public class Product extends Auditable {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    private String name;
    private BigDecimal price;

    @PostLoad
    private void afterLoad() {
        // Called after SELECT — good for computing derived/transient fields
        System.out.println("Loaded product: " + name);
    }

    @PreRemove
    private void beforeDelete() {
        // Called before DELETE — good for validation or cleanup
        System.out.println("Deleting product: " + name);
    }
}

// which works with Spring Security to also capture who made the change

Q19. What is Spring Data JPA Specification? How does it enable dynamic queries?

A derived query method produces a fixed query — you cannot change its WHERE conditions at runtime. If a user can search by any combination of name, category, price range, or status, you would need a separate method for every combination — which becomes unmanageable. Spring Data JPA Specification (based on JPA Criteria API) solves this by letting you build query predicates programmatically and combine them at runtime. Your repository extends JpaSpecificationExecutor, and you pass Specification objects that are built based on which filters the user provided. Each Specification is a small, composable piece of query logic. You combine them with and() and or() to build arbitrarily complex dynamic queries without writing any JPQL strings.
Java 

// Repository must extend JpaSpecificationExecutor
public interface ProductRepository extends JpaRepository,
                                            JpaSpecificationExecutor {}

// Reusable Specification building blocks
public class ProductSpecifications {

    public static Specification hasName(String name) {
        return (root, query, cb) -> name == null ? null
            : cb.like(cb.lower(root.get("name")), "%" + name.toLowerCase() + "%");
    }

    public static Specification hasCategory(String category) {
        return (root, query, cb) -> category == null ? null
            : cb.equal(root.get("category"), category);
    }

    public static Specification priceBetween(BigDecimal min, BigDecimal max) {
        return (root, query, cb) -> {
            if (min == null && max == null) return null;
            if (min == null) return cb.lessThanOrEqualTo(root.get("price"), max);
            if (max == null) return cb.greaterThanOrEqualTo(root.get("price"), min);
            return cb.between(root.get("price"), min, max);
        };
    }
}

// Service — combine specifications based on what the user actually provides
@Service
public class ProductService {

    public List search(String name, String category,
                                BigDecimal minPrice, BigDecimal maxPrice) {
        Specification spec = Specification
            .where(ProductSpecifications.hasName(name))
            .and(ProductSpecifications.hasCategory(category))
            .and(ProductSpecifications.priceBetween(minPrice, maxPrice));

        return productRepository.findAll(spec);
    }
}

Q20. What is the difference between FetchType.LAZY and FetchType.EAGER? What are the common pitfalls with lazy loading outside a transaction?

FetchType.EAGER loads the associated entity immediately in the same query as the parent — you always get it whether you need it or not, which can cause unwanted JOINs and unnecessary data loading. FetchType.LAZY loads the associated entity only when you access the field for the first time — the field is a proxy until then, which is more efficient. The most common pitfall is the LazyInitializationException — this happens when you try to access a lazily loaded association outside of a transaction (and therefore outside of an open persistence context). For example, loading an entity in a @Transactional service method, returning it to the controller, and then accessing a lazy collection in the controller or view — by that point the session is closed. Solutions are: use JOIN FETCH or @EntityGraph to load what you need within the transaction, use DTO projections, or use the Open Session in View pattern (enabled by default in Spring Boot but discouraged in production).
Java 

@Entity
public class Order {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    // LAZY — default for @OneToMany, loaded only when accessed
    @OneToMany(mappedBy = "order", fetch = FetchType.LAZY)
    private List items;

    // EAGER — default for @ManyToOne, always loaded
    @ManyToOne(fetch = FetchType.EAGER)  // consider changing to LAZY
    @JoinColumn(name = "customer_id")
    private Customer customer;
}

// PITFALL: LazyInitializationException
@Service
@Transactional
public class OrderService {
    public Order findById(Long id) {
        return orderRepository.findById(id).orElseThrow();
    }
}

@RestController
public class OrderController {
    public OrderResponse getOrder(Long id) {
        Order order = orderService.findById(id);  // transaction ends here
        // items is a LAZY proxy — session is CLOSED
        order.getItems().size();    // LazyInitializationException!
    }
}

// FIX 1: JOIN FETCH — load within the transaction
@Query("SELECT o FROM Order o JOIN FETCH o.items WHERE o.id = :id")
Optional findByIdWithItems(@Param("id") Long id);

// FIX 2: @EntityGraph
@EntityGraph(attributePaths = {"items", "customer"})
Optional findById(Long id);

// FIX 3: return a DTO from the service — no lazy associations exposed
public record OrderDTO(Long id, String customerName, int itemCount) {}

@Transactional(readOnly = true)
public OrderDTO findById(Long id) {
    Order o = orderRepository.findByIdWithItems(id).orElseThrow();
    return new OrderDTO(o.getId(), o.getCustomer().getName(), o.getItems().size());
}
