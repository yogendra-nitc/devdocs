Interview questions on Generics, Lambda Expressions, Streams and Other Java 8 Features

Q1. What are generics in Java and why were they introduced? What problem did they solve before Java 5?

Before Java 5, collections stored everything as Object. You had to cast manually every time you retrieved an element, and the compiler couldn't catch type mistakes — a ClassCastException would only show up at runtime. Generics let you parameterise types at compile time. The compiler enforces that only the declared type goes in and comes out, catching bugs early. The type information is erased at runtime (type erasure), so generics have zero overhead at the JVM level.
Java 

    // Before generics (Java 1.4) — unsafe, error-prone
    List list = new ArrayList();
    list.add("hello");
    list.add(42);              // compiles fine — no type safety
    String s = (String) list.get(1);  // ClassCastException at RUNTIME

    // With generics (Java 5+) — compiler catches the mistake
    List names = new ArrayList<>();
    names.add("hello");
    names.add(42);             // COMPILE ERROR — caught before running
    String name = names.get(0); // no cast needed

Q2. What is a generic class and a generic method? How do you define and use them?

A generic class has a type parameter in its declaration (e.g., class Box), making the class reusable for any type T without duplicating code. A generic method has its own type parameter independent of the class, declared before the return type. This is useful for utility methods in non-generic classes. The actual type is inferred by the compiler from the argument — you rarely need to specify it explicitly.
Java 

    // Generic class — works for any type T
    class Box {
        private T value;
        public Box(T value) { this.value = value; }
        public T get()      { return value; }
    }

    Box  strBox = new Box<>("Hello");
    Box intBox = new Box<>(100);

    String s = strBox.get();   // no cast
    int    n = intBox.get();   // auto-unboxed

    // Generic method — type parameter on the method itself
    public static  List repeat(T item, int times) {
        List result = new ArrayList<>();
        for (int i = 0; i < times; i++) result.add(item);
        return result;
    }

    List words = repeat("Java", 3);   // ["Java","Java","Java"]
    List nums = repeat(7, 4);        // [7,7,7,7]

Q3. What are bounded type parameters? Explain upper-bounded and lower-bounded wildcards with examples.

Bounded type parameters restrict which types can be used as a type argument. An upper bound () means T must be Number or a subclass of it. A lower bound () means the wildcard can be Integer or any ancestor. The PECS rule helps remember when to use each: Producer Extends, Consumer Super. If a method produces (returns) values from a collection, use extends. If it consumes (adds to) a collection, use super.
Java 

    // Upper bound: T must be Number or subclass (Integer, Double…)
    public static  double sum(List list) {
        double total = 0;
        for (T item : list) total += item.doubleValue();
        return total;
    }
    sum(List.of(1, 2, 3));          // works — Integer extends Number
    sum(List.of(1.5, 2.5));        // works — Double extends Number

    // PECS — Producer Extends
    public static void printAll(List shapes) {
        shapes.forEach(System.out::println);  // reading (producing)
    }

    // PECS — Consumer Super
    public static void addCircles(List list) {
        list.add(new Circle());  // writing (consuming)
    }

Q4. What is type erasure in Java generics? What are its implications?

Type erasure means the compiler removes all generic type information at compile time and replaces type parameters with their bounds (or Object if unbounded). The resulting bytecode has no generic type information — the JVM sees raw types. This ensures backward compatibility with pre-Java-5 code. Implications: you cannot use instanceof with generic types, cannot create arrays of generic types, cannot create new instances of type parameters, and cannot catch generic exception types.
Java 

    // What you write:
    List names = new ArrayList();
    names.add("Alice");

    // What the compiler actually compiles to (after erasure):
    List names = new ArrayList();
    names.add("Alice");
    String s = (String) names.get(0);  // compiler inserts the cast for you

    // Implications of type erasure:

    // 1. Cannot use instanceof with generic type
    // if (list instanceof List) { }  // COMPILE ERROR

    // 2. Cannot create generic arrays
    // T[] arr = new T[10];  // COMPILE ERROR

    // 3. Cannot instantiate T directly
    // T obj = new T();  // COMPILE ERROR

    // 4. But this works (unchecked cast warning)
    @SuppressWarnings("unchecked")
    T value = (T) someObject;

Q5. What is the difference between List, List, and List? When would you use each?

List explicitly holds any Object — but a List is NOT a List (no covariance). You cannot pass a List where a List is expected. List is an unbounded wildcard — it accepts any List regardless of its type parameter, but you cannot add anything to it (except null) because the actual type is unknown. List is a type parameter — it lets you write reusable code where the type is determined at the call site and you CAN read and write the same type T.
